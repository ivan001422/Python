
## Что такое валидация?
**Валидация** — это процесс проверки данных на соответствие определённым правилам (формату, типу, диапазону, бизнес-логике) до того, как они будут использованы в программе. Цель — убедиться, что входные данные корректны, безопасны и пригодны для дальнейшей обработки.

---

## Зачем нужна валидация?
- **Безопасность** — предотвращение инъекций, переполнений, ошибок доступа.
- **Целостность данных** — защита от повреждённых или некорректных записей.
- **Стабильность программы** — избежание `TypeError`, `IndexError` и других исключений в рантайме.
- **Удобство пользователя** — возможность дать понятное сообщение об ошибке вместо неожиданного падения.
- **Соответствие бизнес-правилам** — например, дата не может быть в прошлом, возраст не отрицательный.

---

## Когда использовать?
Везде, где программа принимает данные из внешнего источника:
- Пользовательский ввод (консоль, GUI, веб-формы).
- Файлы (CSV, JSON, XML, конфигурации).
- Аргументы командной строки.
- Данные из базы данных (при десериализации).
- Внутренние вызовы функций (чтобы быть уверенным в типах аргументов).

---

## Виды валидации
### По уровню проверки
- **Синтаксическая** — соответствует ли тип, формат (например, email содержит `@`).
- **Семантическая** — имеет ли значение смысл (например, дата не 31 февраля).
- **Бизнес-логика** — удовлетворяет ли данным правилам предметной области (например, возраст ≥ 18).

### По времени выполнения
- **Статическая** (type hints, линтеры) — проверяется до выполнения кода.
- **Динамическая** (runtime) — проверяется в процессе работы программы.

### По месту (не будет разбираться тут)
- **На клиенте** — быстрая обратная связь, но небезопасно.
- **На сервере** — обязательная, основная защита.
- **В БД** — ограничения схемы (NOT NULL, UNIQUE, CHECK).

---

## Как правильно реализовывать валидацию в Python

### Принципы хорошей валидации
- **Единая ответственность** — валидация должна быть отдельной функцией/классом, не смешиваться с бизнес-логикой.
- **Понятные сообщения об ошибках** — сообщать, что именно не так и как исправить.
- **Возвращать результат или выбрасывать исключение** — единообразно.
- **Не изменять данные** (кроме нормализации, например, обрезка пробелов).
- **Валидировать все входные параметры**, даже если они кажутся доверенными.

### Паттерны валидации
#### a. Функция-валидатор с исключениями
```python
def validate_age(age):
    if not isinstance(age, int):
        raise TypeError("Age must be integer")
    if age < 0 or age > 150:
        raise ValueError("Age must be between 0 and 150")
    return age  # или True
```

#### b. Функция, возвращающая булево значение + сообщение
```python
def is_valid_email(email):
    if not isinstance(email, str):
        return False, "Email must be string"
    if "@" not in email:
        return False, "Email must contain @"
    return True, "OK"
```

#### c. Класс-валидатор (для сложных объектов)
```python
class UserValidator:
    def validate(self, user_dict):
        errors = {}
        if 'name' not in user_dict:
            errors['name'] = 'Name is required'
        # ...
        if errors:
            raise ValidationError(errors)
```

### Обратите внимание, порядок проверок валидации идёт сверху вниз (по уровню проверки, описанному выше), проверяя все возможные ошибки.
1) Тип  
2) Семантика  
3) Логика  

Внимательно относитесь к порядку проверок валидации. **Неправильный порядок** приведёт к ошибкам:

```python
def process_age(age):
    age = int(age)        # может упасть, если строка "abc"
    if age < 0: ...       # а до этого уже ошибка
```

Здесь мы сразу проверяем семантику, забывая про тип.

---

### Обработка ошибок валидации
- Используйте собственные классы исключений (например, `ValidationError`).
- Собирайте все ошибки, а не останавливайтесь на первой (для пользователя лучше увидеть сразу все проблемы).

---

## Распространённые ошибки при валидации
- **Смешивание валидации и бизнес-логики**.
- **Использование голого `except:`** — скрывает ошибки.
- **Не проверять типы** — полагаться на то, что данные придут в нужном формате.
- **Сообщения об ошибках неинформативны** (например, просто "Error").
- **Валидировать только часть данных** — забыть о важных полях.

---

## Best Practices
- **Валидируйте данные на границе системы** (при входе).
- **Стремитесь к иммутабельности** — валидация не должна менять данные.
- **Пишите тесты для валидаторов** — проверяйте корректные и некорректные случаи.
- **Выносите сложные правила в отдельные классы или функции**.
- **Для больших проектов используйте Pydantic/attrs** — они автоматизируют многие проверки (про них в каталоге OOP).

# Учимся этикету валидации

## Аккумуляция ошибок (сбор всех проблем за один проход)
Вместо того чтобы прерывать выполнение при первой же ошибке, вы можете собрать все ошибки и вернуть их пользователю. Это улучшает UX.

```python
from typing import List, Tuple

def validate_user(data: dict) -> Tuple[bool, List[str]]:
    errors = []
    
    # 1. Проверка наличия полей
    required = ['name', 'age']
    for field in required:
        if field not in data:
            errors.append(f"Missing required field: {field}")
    
    # 2. Проверка типов (если поле есть)
    if 'name' in data and not isinstance(data['name'], str):
        errors.append("Name must be a string")
    if 'age' in data and not isinstance(data['age'], int):
        errors.append("Age must be an integer")
    
    # 3. Проверка значений (если тип корректен)
    if 'age' in data and isinstance(data['age'], int):
        if data['age'] < 0 or data['age'] > 150:
            errors.append("Age must be between 0 and 150")
    
    # 4. Возврат результата
    return len(errors) == 0, errors
```

## Валидация с нормализацией данных

Иногда нужно не только проверить, но и привести данные к единому формату (например, обрезать пробелы, преобразовать строки в числа). Делайте это **после всех проверок**, чтобы не усложнять валидацию.

```python
class ValidationError(Exception):
    pass

def normalize_and_validate(user_data):
    errors = []
    # валидация
    if 'name' not in user_data or not user_data['name'].strip():
        errors.append("Name is required")
    if 'age' not in user_data:
        errors.append("Age is required")
    else:
        try:
            age = int(user_data['age'])
            if age < 0:
                errors.append("Age must be positive")
        except ValueError:
            errors.append("Age must be a number")
    
    if errors:
        raise ValidationError(errors)
    
    # нормализация (после валидации)
    normalized = {
        'name': user_data['name'].strip(),
        'age': int(user_data['age'])
    }
    return normalized
```

Это неплохо для небольших проектов, но для больших лучше выносить нормализацию отдельно (про валидацию в каталоге OOP).

## Использование try/except для приведения типов

В Python часто приходится пробовать преобразовать строку в число. Используйте try/except только для ожидаемых исключений (`ValueError`, `TypeError`). **Никогда не используйте голый except:** – это скроет ошибки, которые не должны быть скрыты.

```python
def safe_int(value, field_name="value"):
    if isinstance(value, int):
        return value
    if isinstance(value, str):
        try:
            return int(value)
        except ValueError:
            raise ValueError(f"{field_name} must be a valid integer, got '{value}'")
    raise TypeError(f"{field_name} must be int or str")
```

Не забывайте при необходимости создавать собственные исключения.

## Валидация в классах (методы __init__, сеттеры, свойства)

В `__init__` всегда проверяйте аргументы перед присваиванием.
Используйте `@property` для контроля доступа и валидации при установке.

```python
class Person:
    def __init__(self, name, age):
        self.name = name   # вызовет setter
        self.age = age

    @property
    def age(self):
        return self._age

    @age.setter
    def age(self, value):
        if not isinstance(value, int):
            raise TypeError("Age must be int")
        if value < 0:
            raise ValueError("Age cannot be negative")
        self._age = value
```

Сеттеры помогут избежать ошибок присваивания. Делайте проверку валидации внутри них, это может избежать ошибок присваивания.

# Декораторы для валидации в Python

## Что такое декоратор?

**Декоратор** — это функция, которая принимает другую функцию (или класс) и возвращает новую функцию, обычно добавляющую или изменяющую поведение исходной. В Python декораторы часто используются для **аспектно-ориентированного программирования** (AOP), позволяя отделить сквозную логику (логирование, проверка прав, кеширование, **валидация**) от основной бизнес-логики.

---

## Зачем нужны декораторы для валидации?

- **Разделение ответственности** – код валидации не смешивается с основным кодом функции.
- **Переиспользование** – один декоратор можно применить к множеству функций.
- **Читаемость** – декоратор виден сразу над определением функции, ясно, что она выполняет проверки.
- **Гибкость** – можно комбинировать несколько декораторов (например, `@validate_types` + `@log`).
- **Удобство тестирования** – валидацию можно тестировать независимо.

---

## Преимущества использования декораторов для валидации

| Преимущество | Пояснение |
|--------------|-----------|
| **Чистота кода** | Основная функция не содержит проверок, сосредоточена на своей задаче. |
| **DRY (Don't Repeat Yourself)** | Один декоратор применяется везде, где нужна одинаковая валидация. |
| **Прозрачность** | В сигнатуре функции не видно лишних параметров, декоратор «оборачивает» её. |
| **Легкость модификации** | Чтобы изменить правила валидации, достаточно поправить декоратор, а не все функции. |
| **Композиция** | Можно навешивать несколько декораторов, каждый выполняет свою проверку. |

---

## Как правильно создавать декораторы для валидации

Перед началом ознакомьтесь:

### Что такое функция в Python?

Функция — это объект. Вы можете присвоить её переменной, передать как аргумент в другую функцию, вернуть из функции. Например:

```python
def say_hello():
    print("Привет!")

a = say_hello   # теперь a — это та же функция
a()             # выведет "Привет!"
```

---

### Декоратор без синтаксического сахара

Теперь напишем декоратор, который делает проверку x > 0.

Сначала напишем функцию-декоратор (она принимает функцию и возвращает новую):

```python
def positive(func):         # func — это исходная функция
    def wrapper(x):         # wrapper — новая функция, которая будет вызываться
        if x <= 0:
            raise ValueError("Число должно быть положительным")
        return func(x)      # вызываем исходную функцию
    return wrapper          # возвращаем wrapper вместо исходной
```

Теперь у нас есть простая функция `sqrt`:

```python
def sqrt(x):
    return x ** 0.5
```

Если мы хотим добавить проверку на положительность, мы **переопределяем** `sqrt`, заменяя её на обёртку:

```python
sqrt = positive(sqrt)   # теперь sqrt — это не исходная функция, а wrapper
```

Теперь, когда мы вызываем `sqrt(9)`, на самом деле вызывается `wrapper(9)`, который проверяет `x` и, если всё хорошо, вызывает исходную `sqrt`.

Это и есть **переопределение функции**: мы берём существующее имя `sqrt` и присваиваем ему новое значение (обёртку). Исходная функция при этом сохраняется внутри `wrapper` как `func`.

---

### Синтаксический сахар `@positive`

Вместо того чтобы писать две строки (определение и переопределение), Python позволяет сделать это одной конструкцией:

```python
@positive
def sqrt(x):
    return x ** 0.5
```

Это **ровно то же самое**, что:

```python
def sqrt(x):
    return x ** 0.5
sqrt = positive(sqrt)
```

Символ `@` просто говорит интерпретатору: «возьми функцию, которая идёт ниже, передай её в декоратор `positive`, и результат присвой обратно имени этой функции».

---

### Что происходит при вызове?

Когда вы пишете `sqrt(9)`, Python смотрит, что сейчас находится под именем `sqrt`. А там — `wrapper` (потому что мы переопределили `sqrt`). Поэтому вызывается `wrapper(9)`. Внутри `wrapper`:

- Проверяет `x`.
- Если проверка пройдена, вызывает `func(9)`, где `func` — это исходная функция, которая была передана в декоратор. Она сохранена в замыкании (благодаря тому, что `wrapper` определена внутри `positive` и использует `func`).

Это и есть **замыкание**: функция `wrapper` «запоминает» переменную `func` из внешней области видимости.

---

### Зачем нужен `functools.wraps`?

Когда мы пишем `sqrt = positive(sqrt)`, новая функция `wrapper` имеет своё имя (`wrapper`) и свою документацию (если есть). Если мы не используем `wraps`, то:

```python
print(sqrt.__name__)   # выведет "wrapper", а не "sqrt"
help(sqrt)             # покажет документацию wrapper, а не исходной функции
```

Это неудобно для отладки и для автоматической генерации документации. `functools.wraps(func)` — это декоратор, который копирует из `func` в `wrapper` имя, докстринг, аннотации и т.д. Он применяется к `wrapper`:

```python
import functools

def positive(func):
    @functools.wraps(func)      # это декоратор, который обновляет wrapper
    def wrapper(x):
        ...
    return wrapper
```

Теперь `sqrt.__name__` снова станет `"sqrt"`, и документация останется исходной.

### Декоратор с параметрами (конфигурацией)

Позволяет задавать допустимые значения или условия:

```python
import functools

def validate_range(min_val, max_val): # Укажем параметры при добавлении декоратора
    def decorator(func):
        @functools.wraps(func)
        def wrapper(x):
            if not isinstance(x, (int, float)):
                raise TypeError(f"Argument must be number, got {type(x).__name__}")
            if not (min_val <= x <= max_val):
                raise ValueError(f"Argument must be between {min_val} and {max_val}")
            return func(x)
        return wrapper
    return decorator

@validate_range(0, 100)
def set_percent(p):
    return p / 100

print(set_percent(50))   # 0.5
# set_percent(150)       # ValueError
```

**Зачем нужен такой декоратор?**  
Позволяет параметризовать правила валидации, делая декоратор универсальным.

---

## Важные детали при создании декораторов

- **Используйте `functools.wraps`** – без него имя, докстринг и аннотации обёрнутой функции будут потеряны, что ухудшает отладку и документацию.
- **Не меняйте сигнатуру** – декоратор должен принимать те же аргументы, что и исходная функция (или быть универсальным через `*args, **kwargs`).
- **Бросайте конкретные исключения** – `TypeError`, `ValueError` или свои кастомные, чтобы вызывающий код мог их обработать.
- **Не злоупотребляйте** – если валидация очень сложная или требует доступа к БД, лучше вынести её в отдельный сервис, а не в декоратор.
- **Порядок декораторов важен** – декораторы применяются снизу вверх. Валидационный декоратор обычно ставят ближе к функции, чтобы он выполнился первым.

---

## Когда использовать декораторы для валидации, а когда – нет?

| **Когда использовать** | **Когда лучше не использовать** |
|------------------------|--------------------------------|
| - Однотипные проверки на нескольких функциях (типы, диапазоны). | - Валидация сложная, с бизнес-правилами, зависящими от состояния системы. |
| - Проверка только входных аргументов. | - Проверки, которые требуют доступа к внешним ресурсам (БД, API). |
| - Хотите сохранить чистоту основной функции. | - Валидация настолько специфична, что декоратор будет иметь много параметров. |
