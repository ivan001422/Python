## Глава 1: Объектно-ориентированное проектирование

### 1.1. Что такое ООП?

**Объектно-ориентированное программирование** — это способ организации кода, при котором программа строится как совокупность взаимодействующих **объектов**. Каждый объект содержит **данные** (своё состояние) и **поведение** (методы, которые могут изменять это состояние или делать что-то полезное). Такой подход позволяет управлять сложностью больших проектов, разбивая их на логические блоки. Вместо того чтобы держать в голове все детали сразу, вы работаете с отдельными сущностями, каждая из которых решает свою задачу.

**Три этапа ООП** (на практике они переплетаются):  
- **Объектно-ориентированный анализ (ООА)** – выявление объектов и их взаимодействий на основе требований (ответ на вопрос «Что мы хотим получить?»).  
- **Объектно-ориентированное проектирование (ООП)** – спецификация классов, атрибутов, методов и связей между ними («Как мы это сделаем?»).  
- **Объектно-ориентированное программирование** – непосредственная реализация на языке Python.

В реальных проектах эти этапы не следуют строго один за другим, а выполняются итеративно, с постоянным уточнением. Благодаря этому вы можете быстро адаптироваться к новым требованиям и исправлять ошибки на ранних стадиях.

### 1.2. Объекты, классы, атрибуты и методы

- **Класс** – это шаблон (чертёж) для создания объектов. Он описывает, какие атрибуты и методы будут у объектов этого типа.  
- **Объект (экземпляр)** – конкретный представитель класса, обладающий собственным состоянием. Например, все автомобили имеют общие атрибуты (цвет, модель), но у каждого конкретного автомобиля они свои.

**Атрибуты** – переменные, хранящие состояние объекта (например, `вес`, `цвет`).  
**Методы** – функции, определённые внутри класса, которые могут читать или изменять атрибуты, а также выполнять другие действия.

Классы позволяют описать сущности предметной области (пользователи, заказы, товары) и их поведение. Это делает код более выразительным и структурированным – вы работаете с «объектами», а не с сырыми данными и функциями вразнобой. Кроме того, классы облегчают повторное использование: однажды написанный класс можно применять в разных частях программы или даже в других проектах.

В **UML** (унифицированный язык моделирования) классы изображаются прямоугольниками, отношения – стрелками.  
- Атрибуты и методы записываются в секциях класса.  
- Знак `+` означает публичный доступ, `-` – приватный (в Python это соглашение, а не строгое правило). UML помогает визуализировать структуру системы и обсуждать её с коллегами, не углубляясь в код.

---

### 1.3. Отношения между классами

Объекты редко существуют изолированно – они взаимодействуют. Основные виды отношений:

- **Ассоциация** – общая связь (например, «студент посещает курс»).  
- **Композиция** – жёсткая связь: объект-часть не может существовать без целого (например, «двигатель – часть автомобиля»). Обозначается чёрным ромбом.  
- **Агрегация** – более слабая связь: части могут существовать отдельно (например, «список студентов»). Обозначается белым ромбом.

Правильно выбранные отношения между классами позволяют моделировать реальные связи без дублирования данных. Композиция помогает собирать сложные объекты из простых, а агрегация даёт гибкость при работе с коллекциями. Это также упрощает тестирование, так как каждый класс можно проверять независимо.

**Наследование** – отношение **«is-a»** (является). Подкласс наследует все атрибуты и методы суперкласса и может их переопределять. Мы рассмотрим его отдельно ниже.

---

### 1.4. Инкапсуляция и абстракция

- **Инкапсуляция** – скрытие внутренней реализации и предоставление внешнего интерфейса (публичных методов). В Python нет строгой приватности, но есть соглашение: имена с ведущим подчёркиванием `_` считаются «внутренними».  
- **Абстракция** – выделение только тех деталей, которые важны для текущей задачи, игнорирование остальных.

Инкапсуляция защищает объект от некорректного использования. Вы можете менять внутреннюю логику класса, не боясь сломать код, который с ним работает, – главное, чтобы публичный интерфейс оставался стабильным. Это особенно важно в больших командах, где разные разработчики поддерживают разные части системы. Абстракция же помогает упрощать сложные системы, фокусируясь на сути, – например, для пользователя банковского приложения важны только доступные операции, а не внутренние алгоритмы обработки транзакций.

---

### 1.5. Наследование и полиморфизм

- **Наследование** позволяет создавать иерархии классов, повторно используя код родительского класса.  
- **Переопределение методов** – подкласс может изменить поведение метода суперкласса.  
- **Полиморфизм** – возможность работать с объектами разных классов через единый интерфейс (метод с одинаковым именем).  
- В Python полиморфизм часто реализуется через **утиную типизацию**: «Если объект ходит как утка и крякает как утка, то это утка» – не важен класс, важны поддерживаемые методы.

Наследование сокращает дублирование кода и позволяет выразить общие свойства в одном месте – например, базовый класс `Фигура` может содержать метод `площадь()`, а подклассы `Круг` и `Прямоугольник` реализуют его по-своему. Полиморфизм даёт возможность писать универсальные алгоритмы, которые работают с любыми объектами, поддерживающими нужный интерфейс. Это делает систему гибкой и расширяемой: вы можете добавлять новые типы, не изменяя существующий код.

---

### 1.6. Множественное наследование

Класс может наследовать от нескольких родителей. В Python порядок разрешения методов (MRO) определяет, какой метод будет вызван при конфликте.  
MRO можно посмотреть через атрибут `__mro__`.  
При множественном наследовании важно правильно использовать `super()` для вызова методов родительских классов, чтобы избежать повторных вызовов и проблем с аргументами.

Иногда класс должен объединять функциональность из разных источников (например, «летающий автомобиль»). Множественное наследование позволяет собрать такое поведение, но требует осторожности. В Python часто используют **миксины** – небольшие классы-примеси, которые добавляют конкретную возможность (например, `Loggable`, `Serializable`). Это удобный способ расширить класс без создания глубоких иерархий.

### 1.7. Тематическое исследование: классификация ирисов

**Задача**: классифицировать цветки ириса по четырём числовым признакам с помощью алгоритма **k-ближайших соседей (k‑NN)**.  
**Роли**:  
- **Ботаник** – предоставляет размеченные обучающие данные, настраивает гиперпараметры.  
- **Пользователь** – отправляет запрос с неизвестным образцом и получает предсказанный вид.

**Логическое представление** (классы):  
- `TrainingData` – хранит обучающую и тестовую выборки, а также метаданные.  
- `Sample` – содержит четыре измерения (длины и ширины чашелистиков и лепестков).  
- `KnownSample` – расширяет `Sample` добавлением известного вида.  
- `Hyperparameter` – содержит значение `k` и ссылку на `TrainingData`; метод `test()` оценивает качество классификации.

**Процессы**:  
1. Загрузка данных и разделение на обучающую и тестовую части.  
2. Тестирование различных гиперпараметров (значений `k` и алгоритмов расстояния).  
3. Классификация нового образца.

**Программное и физическое представления** – обсуждается разбиение на модули и развёртывание в виде веб-приложения.

Этот пример иллюстрирует, как теоретические концепции (классы, наследование, композиция) применяются в реальной задаче машинного обучения. Вы видите, как проектируются классы, распределяются обязанности и строится система, которую можно тестировать, модифицировать и масштабировать. Такой подход не только помогает решить текущую задачу, но и закладывает основу для дальнейшего развития – например, для добавления других алгоритмов или источников данных.

## Глава 2: Объекты в Python

### 2.1. Подсказки типов (Type Hints)

Python – язык с динамической типизацией, но для улучшения читаемости и раннего обнаружения ошибок можно использовать **аннотации типов**.  

Пример:
```python
def odd(n: int) -> bool:
    return n % 2 != 0
```
- `n: int` – параметр должен быть целым числом.  
- `-> bool` – функция возвращает логическое значение.

Аннотации не влияют на выполнение, но проверяются статическими анализаторами (например, **mypy**). Для коллекций указываются типы элементов: `list[int]`, `dict[str, float]`. Для работы с такими аннотациями рекомендуется импортировать `from __future__ import annotations`.

Аннотации делают код самодокументируемым – сразу видно, какие данные ожидаются и что возвращается. Они помогают IDE давать подсказки и предупреждают о несоответствии типов ещё до запуска программы, экономя время на отладку. Это особенно ценно в больших проектах, где трудно держать в голове все интерфейсы.

---

### 2.2. Создание классов и атрибутов

Класс создаётся ключевым словом `class`:
```python
class MyFirstClass:
    pass
```
Атрибуты можно добавлять динамически:
```python
obj = MyFirstClass()
obj.x = 5
```
Но лучше объявлять их в методе `__init__`.

Классы – это основной инструмент ООП в Python. Они позволяют группировать данные и функции, связанные общей сущностью. Динамическое добавление атрибутов даёт гибкость, но для надёжности стоит инициализировать все атрибуты в `__init__`, чтобы объект всегда был в согласованном состоянии. Это упрощает отладку и делает код предсказуемым для других разработчиков.

---

### 2.3. Методы и параметр self

Метод – это функция, определённая внутри класса. Первый параметр метода всегда **`self`** – ссылка на текущий экземпляр. При вызове `obj.method()` Python автоматически передаёт `obj` как `self`.

Пример:
```python
class Point:
    def reset(self):
        self.x = 0
        self.y = 0
```

Параметр `self` даёт методу доступ к атрибутам конкретного объекта, с которым он работает. Без него метод не смог бы изменять состояние экземпляра. Это центральная идея ООП – методы оперируют данными именно того объекта, для которого они вызваны. Таким образом, один и тот же класс может обслуживать множество объектов, каждый со своим состоянием.

---

### 2.4. Инициализация объектов __init__

Метод `__init__` вызывается при создании объекта. Его используют для задания начальных значений атрибутов.
```python
class Point:
    def __init__(self, x: float = 0, y: float = 0):
        self.x = x
        self.y = y
```
Можно задавать значения по умолчанию, делая параметры необязательными.

`__init__` гарантирует, что каждый новый объект будет иметь все необходимые атрибуты с правильными начальными значениями. Это предотвращает ошибки, связанные с отсутствием атрибутов, и делает код более предсказуемым. Кроме того, конструктор позволяет инкапсулировать логику инициализации, что упрощает создание сложных объектов.

---

### 2.5. Docstrings – строки документации

Первая строка внутри класса или метода, заключённая в тройные кавычки, становится строкой документации. Её можно посмотреть через `help(класс)` или `print(класс.__doc__)`.

Пример:
```python
class Point:
    """Представляет точку в двумерном пространстве."""
    def reset(self):
        """Сбрасывает координаты в (0,0)."""
        self.x = 0
        self.y = 0
```

Docstrings – это встроенная документация, доступная прямо в коде. Они помогают разработчикам быстро понять назначение класса или метода без необходимости читать весь код. Это особенно важно в командной работе и при поддержке проектов, к которым вы возвращаетесь через месяцы. Инструменты вроде Sphinx могут автоматически генерировать документацию из docstrings.

---

### 2.6. Модули и пакеты

- **Модуль** – файл с расширением `.py`. Имя модуля – имя файла.  
- **Пакет** – папка, содержащая файл `__init__.py` (может быть пустым) и другие модули или подпакеты.

Импорт:
```python
import math                     # импорт всего модуля
from math import hypot          # импорт конкретной функции
from database import Database   # импорт класса
```
Использование `from module import *` **не рекомендуется** – это засоряет пространство имён и затрудняет чтение кода.

По мере роста проекта один файл становится слишком большим. Модули позволяют логически разбить код на части, каждая из которых отвечает за свою функциональность. Пакеты добавляют ещё один уровень иерархии, помогая организовывать связанные модули. Это облегчает навигацию и повторное использование кода в других проектах. Кроме того, модули помогают избежать конфликтов имён.

---

### 2.7. Абсолютный и относительный импорт

- **Абсолютный** – указывается полный путь от корневого пакета:
  ```python
  from ecommerce.products import Product
  ```
- **Относительный** – используется внутри пакета для ссылки на соседние модули с помощью точек:
  ```python
  from .database import Database      # текущий пакет
  from ..contact.email import send    # родительский пакет
  ```
Относительный импорт удобен для больших проектов, но злоупотреблять им не стоит – «плоский лучше, чем вложенный».

Абсолютные импорты делают код явным – сразу видно, откуда берётся класс или функция. Относительные импорты упрощают перемещение модулей внутри пакета, так как пути не привязаны к корню проекта. Выбор между ними зависит от структуры и предпочтений команды. Главное – сохранять читаемость и избегать излишней вложенности.

---

### 2.8. Виртуальные окружения и сторонние библиотеки

Для изоляции проектов используются **виртуальные окружения** (модуль `venv`).  
Создание:
```bash
python -m venv env
```
Активация:
- Linux/macOS: `source env/bin/activate`
- Windows: `env\Scripts\activate`

Установка сторонних пакетов через `pip`:
```bash
python -m pip install mypy
```

Разные проекты могут требовать разных версий одних и тех же библиотек. Виртуальные окружения изолируют зависимости, предотвращая конфликты. Кроме того, они позволяют устанавливать пакеты без прав администратора и сохраняют чистоту системного Python. Это стандартная практика в профессиональной разработке, обеспечивающая воспроизводимость сборок.
