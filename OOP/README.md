# Объектно-ориентированное проектирование

## Что такое ООП?

**Объектно-ориентированное программирование (ООП)** — это способ организации кода, при котором программа строится как совокупность взаимодействующих **объектов**. Каждый объект содержит **данные** (своё состояние) и **поведение** (методы, которые могут изменять это состояние или делать что-то полезное). Такой подход позволяет управлять сложностью больших проектов, разбивая их на логические блоки. Вместо того чтобы держать в голове все детали сразу, вы работаете с отдельными сущностями, каждая из которых решает свою задачу.

**Три этапа работы с объектами** (на практике они переплетаются):  

- **Объектно-ориентированный анализ (ООА)** – выявление объектов и их взаимодействий на основе требований («Что мы хотим получить?»).  
- **Объектно-ориентированное проектирование (ООД)** – спецификация классов, атрибутов, методов и связей между ними («Как мы это сделаем?»).  
- **Объектно-ориентированное программирование (ООП)** – непосредственная реализация на языке Python.

В реальных проектах эти этапы не следуют строго один за другим, а выполняются итеративно, с постоянным уточнением. Благодаря этому вы можете быстро адаптироваться к новым требованиям и исправлять ошибки на ранних стадиях.

## Объекты, классы, атрибуты и методы

- **Класс** – это шаблон (чертёж) для создания объектов. Он описывает, какие атрибуты и методы будут у объектов этого типа.  
- **Объект (экземпляр)** – конкретный представитель класса, обладающий собственным состоянием. Например, все автомобили имеют общие атрибуты (цвет, модель), но у каждого конкретного автомобиля они свои.

**Атрибуты** – переменные, хранящие состояние объекта (например, `вес`, `цвет`).  
**Методы** – функции, определённые внутри класса, которые могут читать или изменять атрибуты, а также выполнять другие действия.

Классы позволяют описать сущности предметной области (пользователи, заказы, товары) и их поведение. Это делает код более выразительным и структурированным – вы работаете с «объектами», а не с сырыми данными и функциями вразнобой. Кроме того, классы облегчают повторное использование: однажды написанный класс можно применять в разных частях программы или даже в других проектах.

В **UML** (унифицированный язык моделирования) классы изображаются прямоугольниками, отношения – стрелками.  
- Атрибуты и методы записываются в секциях класса.  
- Знак `+` означает публичный доступ, `-` – приватный (в Python это соглашение, а не строгое правило).  
UML помогает визуализировать структуру системы и обсуждать её с коллегами, не углубляясь в код.

## Отношения между классами

Объекты редко существуют изолированно – они взаимодействуют. Основные виды отношений:

- **Ассоциация** – общая связь (например, «студент посещает курс»).  
- **Композиция** – жёсткая связь: объект-часть не может существовать без целого (например, «двигатель – часть автомобиля»). Обозначается чёрным ромбом.  
- **Агрегация** – более слабая связь: части могут существовать отдельно (например, «университет содержит студентов», но студенты могут существовать и без университета). Обозначается белым ромбом.

Правильно выбранные отношения между классами позволяют моделировать реальные связи без дублирования данных. Композиция помогает собирать сложные объекты из простых, а агрегация даёт гибкость при работе с коллекциями. Это также упрощает тестирование, так как каждый класс можно проверять независимо.

**Наследование** – отношение **«is-a»** (является). Подкласс наследует все атрибуты и методы суперкласса и может их переопределять. Мы рассмотрим его отдельно ниже.

## Инкапсуляция и абстракция

- **Инкапсуляция** – скрытие внутренней реализации и предоставление внешнего интерфейса (публичных методов). В Python нет строгой приватности, но есть соглашение: имена с ведущим подчёркиванием `_` считаются «внутренними».  
- **Абстракция** – выделение только тех деталей, которые важны для текущей задачи, и игнорирование остальных.

Инкапсуляция защищает объект от некорректного использования. Вы можете менять внутреннюю логику класса, не боясь сломать код, который с ним работает, – главное, чтобы публичный интерфейс оставался стабильным. Это особенно важно в больших командах, где разные разработчики поддерживают разные части системы. Абстракция же помогает упрощать сложные системы, фокусируясь на сути, – например, для пользователя банковского приложения важны только доступные операции, а не внутренние алгоритмы обработки транзакций.

## Наследование и полиморфизм

- **Наследование** позволяет создавать иерархии классов, повторно используя код родительского класса.  
- **Полиморфизм** – возможность работать с объектами разных классов через единый интерфейс (метод с одинаковым именем).  

В Python полиморфизм часто реализуется через **утиную типизацию**: «Если объект ходит как утка и крякает как утка, то это утка» – не важен класс, важны поддерживаемые методы.

Наследование сокращает дублирование кода и позволяет выразить общие свойства в одном месте – например, базовый класс `Фигура` может содержать метод `площадь()`, а подклассы `Круг` и `Прямоугольник` реализуют его по-своему. Полиморфизм даёт возможность писать универсальные алгоритмы, которые работают с любыми объектами, поддерживающими нужный интерфейс. Это делает систему гибкой и расширяемой: вы можете добавлять новые типы, не изменяя существующий код.

---

## Множественное наследование (краткое введение)

Класс может наследовать от нескольких родителей. В Python порядок разрешения методов (MRO) определяет, какой метод будет вызван при конфликте.  

Иногда класс должен объединять функциональность из разных источников (например, «летающий автомобиль»). Множественное наследование позволяет собрать такое поведение, но требует осторожности. В Python часто используют **миксины** – небольшие классы-примеси, которые добавляют конкретную возможность (например, `Loggable`, `Serializable`). Это удобный способ расширить класс без создания глубоких иерархий. Подробно мы рассмотрим эту тему позже.

## Объекты в Python

### Подсказки типов (Type Hints)

Python – язык с динамической типизацией, но для улучшения читаемости и раннего обнаружения ошибок можно использовать **аннотации типов**.  

Пример:
```python
def odd(n: int) -> bool:
    return n % 2 != 0
```
- `n: int` – параметр должен быть целым числом.  
- `-> bool` – функция возвращает логическое значение.

Аннотации не влияют на выполнение, но проверяются статическими анализаторами (например, **mypy**). Для коллекций указываются типы элементов: `list[int]`, `dict[str, float]`. Для работы с такими аннотациями рекомендуется импортировать `from __future__ import annotations`.

Аннотации делают код самодокументируемым – сразу видно, какие данные ожидаются и что возвращается. Они помогают IDE давать подсказки и предупреждают о несоответствии типов ещё до запуска программы, экономя время на отладку. Это особенно ценно в больших проектах, где трудно держать в голове все интерфейсы.

---

### Что такое и зачем нужен mypy?

**Mypy** — это статический анализатор типов для Python. Он проверяет ваш код до его запуска и ищет ошибки, связанные с типами данных. Вы расставляете подсказки типов, а mypy следит за их соблюдением.

Пример:
```python
def greet(name: str) -> str:
    return f"Привет, {name}!"

# Случайно передаём число – Python не выдаст ошибку при запуске
greet(123)   # выведет "Привет, 123!"
```
Без mypy такое может привести к неожиданному поведению. Запускаем проверку:
```bash
mypy script.py
```
Получаем ошибку:
```
error: Argument 1 to "greet" has incompatible type "int"; expected "str"
```
Вы исправляете ошибку до того, как код попадёт в продакшен, и экономите часы отладки.

### Создание классов и атрибутов

Класс создаётся ключевым словом `class`:
```python
class MyFirstClass:
    pass
```
Атрибуты можно добавлять динамически:
```python
obj = MyFirstClass()
obj.x = 5
```
Но лучше объявлять их в методе `__init__`.

Классы – это основной инструмент ООП в Python. Они позволяют группировать данные и функции, связанные общей сущностью. Динамическое добавление атрибутов даёт гибкость, но для надёжности стоит инициализировать все атрибуты в `__init__`, чтобы объект всегда был в согласованном состоянии. Это упрощает отладку и делает код предсказуемым для других разработчиков.

---

### Методы и параметр `self`

Метод – это функция, определённая внутри класса. Первый параметр метода всегда **`self`** – ссылка на текущий экземпляр. При вызове `obj.method()` Python автоматически передаёт `obj` как `self`.

Пример:
```python
class Point:
    def reset(self):
        self.x = 0
        self.y = 0
```

Параметр `self` даёт методу доступ к атрибутам конкретного объекта, с которым он работает. Без него метод не смог бы изменять состояние экземпляра. Это центральная идея ООП – методы оперируют данными именно того объекта, для которого они вызваны. Таким образом, один и тот же класс может обслуживать множество объектов, каждый со своим состоянием.

---

### Инициализация объектов `__init__`

Метод `__init__` вызывается при создании объекта. Его используют для задания начальных значений атрибутов.
```python
class Point:
    def __init__(self, x: float = 0, y: float = 0):
        self.x = x
        self.y = y
```
Можно задавать значения по умолчанию, делая параметры необязательными.

`__init__` гарантирует, что каждый новый объект будет иметь все необходимые атрибуты с правильными начальными значениями. Это предотвращает ошибки, связанные с отсутствием атрибутов, и делает код более предсказуемым. Кроме того, конструктор позволяет инкапсулировать логику инициализации, что упрощает создание сложных объектов.

---

### Docstrings – строки документации

Первая строка внутри класса или метода, заключённая в тройные кавычки, становится строкой документации. Её можно посмотреть через `help(класс)` или `print(класс.__doc__)`.

Пример:
```python
class Point:
    """Представляет точку в двумерном пространстве."""
    def reset(self):
        """Сбрасывает координаты в (0,0)."""
        self.x = 0
        self.y = 0
```

Docstrings – это встроенная документация, доступная прямо в коде. Они помогают разработчикам быстро понять назначение класса или метода без необходимости читать весь код. Это особенно важно в командной работе и при поддержке проектов, к которым вы возвращаетесь через месяцы. Инструменты вроде Sphinx могут автоматически генерировать документацию из docstrings.

### Модули и пакеты

- **Модуль** – файл с расширением `.py`. Имя модуля – имя файла.  
- **Пакет** – папка, содержащая файл `__init__.py` (может быть пустым) и другие модули или подпакеты.

Импорт:
```python
import math                     # импорт всего модуля
from math import hypot          # импорт конкретной функции
from database import Database   # импорт класса
```
Использование `from module import *` **не рекомендуется** – это засоряет пространство имён и затрудняет чтение кода.

По мере роста проекта один файл становится слишком большим. Модули позволяют логически разбить код на части, каждая из которых отвечает за свою функциональность. Пакеты добавляют ещё один уровень иерархии, помогая организовывать связанные модули. Это облегчает навигацию и повторное использование кода в других проектах. Кроме того, модули помогают избежать конфликтов имён.

---

### Абсолютный и относительный импорт

- **Абсолютный** – указывается полный путь от корневого пакета:
  ```python
  from ecommerce.products import Product
  ```
- **Относительный** – используется внутри пакета для ссылки на соседние модули с помощью точек:
  ```python
  from .database import Database      # текущий пакет
  from ..contact.email import send    # родительский пакет
  ```
Относительный импорт удобен для больших проектов, но злоупотреблять им не стоит – «плоский лучше, чем вложенный».

Абсолютные импорты делают код явным – сразу видно, откуда берётся класс или функция. Относительные импорты упрощают перемещение модулей внутри пакета, так как пути не привязаны к корню проекта. Выбор между ними зависит от структуры и предпочтений команды. Главное – сохранять читаемость и избегать излишней вложенности.

### Магические методы `__repr__` и `__str__`

В Python каждый класс может определить специальные методы для строкового представления.

- **`__repr__`** – возвращает строку, которая однозначно идентифицирует объект. Обычно содержит всю важную информацию, чтобы разработчик мог понять состояние объекта, а в идеале – воссоздать его.
- **`__str__`** – возвращает «красивое», удобочитаемое представление для конечного пользователя.

Если метод не определён явно, он наследуется от `object`.

**Какой метод когда вызывается:**

| Ситуация | Используемый метод |
|----------|-------------------|
| `print(obj)`, `f"{obj}"` | Сначала `__str__`, если нет — `__repr__` |
| `repr(obj)`, коллекции `[obj]`, `f"{obj!r}"` | Всегда `__repr__` |

Синтаксис `!r` внутри f-строк явно говорит Python: «подставь сюда результат `repr(obj)`». Аналогично `!s` вызывает `str(obj)`.

| Синтаксис | Действие | Эквивалент |
|-----------|----------|------------|
| `{obj!r}` | Вызывает `repr(obj)` | `repr(obj)` |
| `{obj!s}` | Вызывает `str(obj)` | `str(obj)` |

## Наследование. Базовые понятия

Наследование позволяет дочернему классу повторно использовать код родительского (базового) класса. Все классы в Python являются подклассами встроенного класса `object`. Этот класс предоставляет базовые методы и метаданные, необходимые для единообразной работы со всеми объектами.

Если наследование не указано явно, класс автоматически наследуется от `object`. Однако можно указать это явно (в Python 3 это не обязательно):
```python
class MySubClass(object):
    pass
```
Суперклассы (родительские классы) — это классы, от которых наследуются данные и методы. Подкласс (дочерний класс) наследуется от суперкласса, расширяя или изменяя его поведение. Имя родительского класса указывается в круглых скобках после имени подкласса.

### Пример: класс `Contact`

Рассмотрим менеджер контактов, который отслеживает имена и адреса электронной почты. Класс `Contact` отвечает за поддержание глобального списка всех контактов и инициализацию отдельного контакта:

```python
from __future__ import annotations

class Contact:
    all_contacts: list["Contact"] = []

    def __init__(self, name: str, email: str) -> None:
        self.name = name
        self.email = email
        Contact.all_contacts.append(self)

    def __repr__(self) -> str:
        return (
            f"{self.__class__.__name__}("
            f"{self.name!r}, {self.email!r}"
            f")"
        )
```

**Переменные класса:** `all_contacts` — общий список для всех экземпляров, так как определён в теле класса (а не в `__init__`). Доступ к нему можно получить как `Contact.all_contacts` или через `self.all_contacts` (если у экземпляра нет собственного атрибута с таким же именем). **Важно:** при присваивании `self.all_contacts = ...` вы создадите новую переменную экземпляра, а не измените переменную класса. Поэтому для ясности лучше обращаться к переменной класса через имя класса.

```python
c_1 = Contact("Dusty", "dusty@example.com")
c_2 = Contact("Steve", "steve@itmaybeahack.com")
print(Contact.all_contacts)
# [Contact('Dusty', 'dusty@example.com'), Contact('Steve', 'steve@itmaybeahack.com')]
```

### Добавление функциональности через наследование: класс `Supplier`

Допустим, некоторые контакты – поставщики, у которых мы можем заказывать товары. Вместо того чтобы добавлять метод `order()` в `Contact` (что дало бы его всем, даже друзьям), мы создаём отдельный класс `Supplier`, наследующий `Contact`:

```python
class Supplier(Contact):
    def order(self, product: str) -> None:
        print(f"Заказ {product} у {self.name} (email: {self.email})")
```

Теперь объекты `Supplier` имеют все поля и методы `Contact` плюс свой метод `order`:
```python
s = Supplier("Иван", "ivan@supplier.ru")
s.order("Канцелярские товары")   # Заказ Канцелярские товары у Иван (email: ivan@supplier.ru)
```

---

### Переопределение методов и `super()`

Если мы хотим, чтобы у `Supplier` был дополнительный атрибут `company_name`, нужно переопределить `__init__`. **Неправильный подход** (показан только для иллюстрации проблемы):

```python
class Supplier(Contact):
    def __init__(self, company_name: str) -> None:
        self.company_name = company_name
# При попытке Supplier("Иван", "ivan@mail.ru", "ООО Ромашка") возникнет ошибка,
# потому что __init__ принимает только company_name.
```

Из-за переопределения мы теряем инициализацию `name`, `email` и добавление в `all_contacts`. Чтобы **добавить** логику, а не заменить её полностью, используется `super()`, который вызывает метод родительского класса:

```python
class Supplier(Contact):
    def __init__(self, name: str, email: str, company_name: str) -> None:
        super().__init__(name, email)   # вызываем родительский __init__
        self.company_name = company_name
```

Теперь всё работает корректно:
```python
s = Supplier("Иван", "ivan@mail.ru", "ООО Ромашка")
print(s.company_name)   # ООО Ромашка
```

`super()` возвращает объект-посредник, который делегирует вызов следующему классу в цепочке наследования (MRO). Это избавляет от дублирования кода и делает иерархию более устойчивой к изменениям.

---

### Переменные класса и осторожность с `self`

Как упоминалось выше, при чтении `self.all_contacts` Python сначала ищет атрибут в экземпляре, а если не находит – в классе. Однако запись `self.all_contacts = ...` всегда создаёт новый атрибут экземпляра, «затеняя» переменную класса. Поэтому для изменения общей переменной класса всегда используйте `ClassName.attribute`.

## Наследование от встроенных типов

Python позволяет создавать классы, расширяющие встроенные типы (`list`, `dict`, `str` и др.). Это удобно для добавления специализированных методов.

### Расширение `list`: `ContactList`

Создадим список, который умеет искать контакты по имени:

```python
from __future__ import annotations

class ContactList(list["Contact"]):
    def search(self, name: str) -> list["Contact"]:
        matching_contacts: list["Contact"] = []
        for contact in self:
            if name in contact.name:
                matching_contacts.append(contact)
        return matching_contacts
```

Теперь используем его в классе `Contact` вместо обычного списка:
```python
class Contact:
    all_contacts = ContactList()   # расширенный список

    # ... (остальной код без изменений)
```

Поиск становится тривиальным:
```python
print(Contact.all_contacts.search("John"))
```

Такой подход инкапсулирует поведение, связанное со списком контактов, прямо в самом списке, вместо разбросанных функций. Мы получаем все стандартные методы списка (`append`, `pop` и т.д.) плюс нашу собственную логику.

### Расширение `dict`: `LongNameDict`

Аналогично можно расширить словарь, добавив метод поиска самого длинного ключа:

```python
from typing import Optional

class LongNameDict(dict[str, int]):
    def longest_key(self) -> Optional[str]:
        longest = None
        for key in self:
            if longest is None or len(key) > len(longest):
                longest = key
        return longest
```

`Optional[str]` означает, что метод может вернуть `str` или `None` (если словарь пуст). Это помогает mypy проверять корректность обработки результата.

## Подробнее об аннотациях типов

Аннотации типов — синтаксис для указания ожидаемых типов переменных, параметров и возвращаемых значений. Они не влияют на выполнение, но используются статическими анализаторами.

### Проблема прямых ссылок и `from __future__ import annotations`

По умолчанию Python вычисляет аннотации в момент определения функции или класса. Это приводит к ошибкам при ссылке класса на самого себя:

```python
class A:
    def f(self) -> A:   # NameError: имя 'A' ещё не определено
        pass
```

Решение — импортировать `from __future__ import annotations` в начале файла. Тогда аннотации сохраняются как строки и вычисляются позже, что позволяет ссылаться на ещё не определённые имена:

```python
from __future__ import annotations

class A:
    def f(self) -> A:   # теперь работает без кавычек
        pass
```

Это особенно полезно при циклических ссылках и улучшает читаемость.

### Модуль `typing`

`typing` предоставляет дополнительные типы для аннотаций: `List`, `Dict`, `Optional`, `Union`, `Protocol` и другие.

| Тип | Описание |
|-----|----------|
| `List[T]` | Список элементов типа `T` |
| `Dict[K, V]` | Словарь с ключами типа `K` и значениями типа `V` |
| `Optional[T]` | `T` или `None` |
| `Union[T1, T2]` | Либо `T1`, либо `T2` |
| `Protocol` | Описывает структурный контракт (какие методы/атрибуты должны быть у объекта) |

`from __future__ import annotations` и `typing` дополняют друг друга: первый даёт гибкость в использовании имён, второй — инструменты для описания сложных типов.

## Множественное наследование и миксины

Когда классу нужно объединить поведение из нескольких независимых источников, применяется множественное наследование. В Python его часто реализуют через **миксины** – небольшие классы, добавляющие одну конкретную функцию.

### Пример миксина: `MailSender`

```python
class MailSender:
    def send_mail(self, message: str) -> None:
        # Предполагаем, что у объекта есть self.email
        print(f"Отправка письма на {self.email}: {message}")
```

Миксин сам по себе бесполезен – он рассчитывает, что класс-«хозяин» предоставит атрибут `email`. Создадим класс, наследующий и от `Contact`, и от `MailSender`:

```python
class EmailableContact(Contact, MailSender):
    pass
```

Теперь объект обладает всеми методами `Contact` и умеет отправлять письма:
```python
e = EmailableContact("Джон", "john@example.com")
e.send_mail("Привет!")
```

### Протоколы для статической проверки

Чтобы mypy не ругался на `self.email` в `MailSender`, можно описать контракт с помощью `Protocol`:

```python
from typing import Protocol

class Emailable(Protocol):
    email: str

class MailSender(Emailable):
    def send_mail(self, message: str) -> None:
        print(f"Отправка на {self.email}: {message}")
```

Теперь mypy знает, что `MailSender` будет использоваться только с классами, имеющими атрибут `email`.

### Альтернативы множественному наследованию

- **Добавление метода прямо в `Contact`:** нарушает принцип единственной ответственности, метод появляется у всех контактов.
- **Отдельная функция:** теряется инкапсуляция, объект сам не знает, как отправить письмо.
- **Композиция:** объект хранит экземпляр `MailSender` и делегирует ему вызов – больше кода, но иногда оправдано.
- **Monkey patching:** динамическое добавление метода в класс – опасно и усложняет отладку.

Вывод: миксины с множественным наследованием дают гибкий и удобный способ точечного расширения функциональности.

## Композиция против множественного наследования: пример с адресом

Добавим в класс `Friend` домашний адрес. Рассмотрим три подхода.

### Вариант 1. Добавить поля напрямую

```python
class Friend(Contact):
    def __init__(self, name, email, phone, street, city, state, code):
        super().__init__(name, email)
        self.phone = phone
        self.street = street
        self.city = city
        self.state = state
        self.code = code
```

**Плюсы:** быстро.
**Минусы:** дублирование при использовании адреса в других классах, нельзя добавить методы работы с адресом.

### Вариант 2. Композиция – отдельный класс `Address`

```python
class Address:
    def __init__(self, street: str, city: str, state: str, code: str) -> None:
        self.street = street
        self.city = city
        self.state = state
        self.code = code

    def full_address(self) -> str:
        return f"{self.street}, {self.city}, {self.state} {self.code}"

class Friend(Contact):
    def __init__(self, name: str, email: str, phone: str, address: Address) -> None:
        super().__init__(name, email)
        self.phone = phone
        self.address = address
```

Использование:
```python
addr = Address("123 Main St", "New York", "NY", "10001")
friend = Friend("Анна", "anna@mail.ru", "555-1234", addr)
print(friend.address.full_address())
```

**Плюсы:** переиспользование, инкапсуляция, лёгкость добавления методов. Отношение «Friend has-a Address» логично и гибко.

### Вариант 3. Множественное наследование с `AddressHolder`

```python
class AddressHolder:
    def __init__(self, street, city, state, code):
        self.street = street
        self.city = city
        self.state = state
        self.code = code

class Friend(Contact, AddressHolder):
    def __init__(self, name, email, phone, street, city, state, code):
        # проблема: как вызвать оба __init__ с разными параметрами?
        super().__init__(name, email)   # вызовет только один из родителей
        # необходим явный вызов второго родителя, что ломает преимущества super()
```

При разных сигнатурах конструкторов множественное наследование становится громоздким и ненадёжным. Композиция здесь явно предпочтительнее.

## Ромбовидное наследование и порядок разрешения методов (MRO)

**Ромбовидное наследование** возникает, когда два родительских класса имеют общего предка, а дочерний класс наследует от обоих:

```
       BaseClass
       /        \
LeftSubclass  RightSubclass
       \        /
      Subclass
```

### Проблема «наивного» подхода

Если каждый класс в цепочке явно вызывает метод родителя по имени, метод базового класса может выполниться несколько раз:

```python
class BaseClass:
    num_base_calls = 0
    def call_me(self):
        print("Calling method on BaseClass")
        self.num_base_calls += 1

class LeftSubclass(BaseClass):
    num_left_calls = 0
    def call_me(self):
        BaseClass.call_me(self)
        print("Calling method on LeftSubclass")
        self.num_left_calls += 1

class RightSubclass(BaseClass):
    num_right_calls = 0
    def call_me(self):
        BaseClass.call_me(self)
        print("Calling method on RightSubclass")
        self.num_right_calls += 1

class Subclass(LeftSubclass, RightSubclass):
    num_sub_calls = 0
    def call_me(self):
        LeftSubclass.call_me(self)
        RightSubclass.call_me(self)
        print("Calling method on Subclass")
        self.num_sub_calls += 1
```

Вызов `Subclass().call_me()` выведет:
```
Calling method on BaseClass
Calling method on LeftSubclass
Calling method on BaseClass
Calling method on RightSubclass
Calling method on Subclass
```
`BaseClass.call_me` выполнен дважды — это нарушает целостность.

### Решение: `super()` и MRO

Python вычисляет **порядок разрешения методов (MRO)** по алгоритму C3-линеаризации. MRO гарантирует, что:
- подкласс предшествует родителю,
- сохраняется порядок, указанный при наследовании,
- общий предок появляется только один раз и в конце.

Для `Subclass(LeftSubclass, RightSubclass)` MRO:  
`Subclass -> LeftSubclass -> RightSubclass -> BaseClass -> object`

Перепишем код с использованием `super()`:

```python
class BaseClass:
    num_base_calls = 0
    def call_me(self):
        print("Calling method on BaseClass")
        self.num_base_calls += 1

class LeftSubclass(BaseClass):
    num_left_calls = 0
    def call_me(self):
        super().call_me()
        print("Calling method on LeftSubclass")
        self.num_left_calls += 1

class RightSubclass(BaseClass):
    num_right_calls = 0
    def call_me(self):
        super().call_me()
        print("Calling method on RightSubclass")
        self.num_right_calls += 1

class Subclass(LeftSubclass, RightSubclass):
    num_sub_calls = 0
    def call_me(self):
        super().call_me()
        print("Calling method on Subclass")
        self.num_sub_calls += 1
```

Теперь вызов проходит по цепочке MRO и каждый метод срабатывает ровно один раз. Вывод программы:

```
Calling method on BaseClass
Calling method on RightSubclass
Calling method on LeftSubclass
Calling method on Subclass
```

Порядок кажется обратным, потому что печать происходит *после* вызова `super()`, когда управление возвращается от самых глубоких классов.

### Как узнать MRO?

```python
print(Subclass.__mro__)
# или
print(Subclass.mro())
```

**Ключевой принцип:** если все классы в ромбовидной иерархии используют `super()` и имеют совместимые сигнатуры (например, через `*args, **kwargs`), множественное наследование работает предсказуемо и безопасно. В противном случае лучше применять композицию или явные вызовы конкретных родителей.
