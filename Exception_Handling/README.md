### Обработка исключений (try/except)

Блок `try` содержит код, который может вызвать исключение. Если исключение возникает, управление передаётся соответствующему блоку `except`. Если исключение не возникает, блок `except` пропускается.

Синтаксис:
```python
try:
    # потенциально опасный код
except SomeException:
    # обработка ошибки
```

Можно указать несколько типов исключений в одном `except`:
```python
except (ZeroDivisionError, TypeError):
    # общий обработчик
```

Можно использовать несколько блоков `except` для разных типов:
```python
try:
    ...
except ZeroDivisionError:
    print("Деление на ноль")
except ValueError:
    print("Неверное значение")
except Exception as e:
    print(f"Неизвестная ошибка: {e}")
```

**Важно:** порядок блоков имеет значение. Если сначала поставить `except Exception`, то он перехватит все исключения, и более конкретные блоки никогда не сработают. Поэтому сначала указывают более специфичные исключения, а в конце – общие.

Для получения объекта исключения используется ключевое слово `as`:
```python
except ValueError as ex:
    print(f"Ошибка: {ex.args}")
```

### Блоки else и finally

- **`else`** – выполняется, если блок `try` завершился без исключений. Это позволяет отделить «нормальный» код от обработки ошибок.
- **`finally`** – выполняется всегда, независимо от того, возникло исключение или нет. Используется для освобождения ресурсов (закрытие файлов, соединений и т.д.).

Пример:
```python
try:
    f = open("file.txt")
    data = f.read()
except IOError:
    print("Ошибка чтения")
else:
    print("Файл прочитан успешно")
finally:
    f.close()   # всегда закрываем файл
```

Блок `finally` выполняется даже после `return` внутри `try` или `except`. Это гарантирует, что ресурсы будут освобождены.

### Иерархия исключений

Все исключения наследуются от `BaseException`. От него напрямую наследуются:
- `Exception` – базовый класс для всех обычных ошибок (все ошибки, которые мы обычно перехватываем).
- `SystemExit` – вызывается при завершении программы через `sys.exit()`.
- `KeyboardInterrupt` – вызывается при нажатии Ctrl+C.

`SystemExit` и `KeyboardInterrupt` **не являются** подклассами `Exception`. Это сделано специально, чтобы блок `except Exception:` не перехватывал их случайно. Если бы они были подклассами `Exception`, то любой код с `except Exception` перехватил бы и запрос на завершение программы, что помешало бы корректному выходу.

**Правило:** никогда не используйте пустой `except:` (без указания типа). Он перехватывает абсолютно всё, включая `SystemExit` и `KeyboardInterrupt`, что может сделать программу неуправляемой. Вместо этого всегда указывайте либо конкретные типы, либо `except Exception:` (если нужно перехватывать все обычные ошибки).

### Создание собственных исключений

Часто встроенных исключений недостаточно, потому что они не несут предметного смысла. Например, `ValueError` слишком общий – он не отличает «недостаточно средств» от «неверный формат номера счёта». Создавая своё исключение, мы даём ему осмысленное имя и можем передавать дополнительную информацию (баланс, запрошенную сумму и т.д.).
Для создания собственного исключения достаточно унаследовать от `Exception` (или от подходящего подкласса). Можно переопределить `__init__`, чтобы принимать нужные параметры, и добавить методы для удобства.

Пример(банковское приложение):
```python
from decimal import Decimal

class InvalidWithdrawal(ValueError):
    def __init__(self, balance: Decimal, amount: Decimal):
        super().__init__(f"Недостаточно средств: баланс {balance}, запрошено {amount}")
        self.balance = balance
        self.amount = amount

    def overdraft(self) -> Decimal:
        return self.amount - self.balance
```

Использование:
```python
try:
    balance = Decimal('25.00')
    raise InvalidWithdrawal(balance, Decimal('50.00'))
except InvalidWithdrawal as ex:
    print(f"Превышение: {ex.overdraft()}")
```

Преимущества:
- **Ясность** – имя исключения сразу говорит о проблеме.
- **Контекст** – обработчик может получить доступ к балансу и запрошенной сумме.
- **Гибкость** – можно добавить методы для вычислений или логирования.

При проектировании библиотек или API собственные исключения делают интерфейс более понятным и удобным для клиентского кода.

### EAFP vs LBYL

В Python принят подход **EAFP** – «Easier to Ask for Forgiveness than Permission» (проще попросить прощения, чем разрешения). Это означает, что мы пытаемся выполнить действие, а если возникает исключение, обрабатываем его. Альтернатива – **LBYL** – «Look Before You Leap» (посмотри, прежде чем прыгнуть), то есть сначала проверяем условия, потом выполняем.

Пример для деления:
- LBYL: `if divisor != 0: result = 100 / divisor`
- EAFP: `try: result = 100 / divisor except ZeroDivisionError: ...`

Почему EAFP предпочтительнее в Python?
- **Производительность** – в случае успеха блок `try` работает быстрее, чем проверка `if` (особенно если проверка сложная).
- **Безопасность** – в многопоточном окружении между проверкой и действием состояние может измениться (гонка). Исключение перехватывается атомарно.
- **Читаемость** – код часто становится короче и понятнее, особенно если нужно выполнить много операций с возможными ошибками.

Однако не стоит злоупотреблять EAFP в горячих циклах, где исключения могут возникать часто – создание и перехват исключений дорого. Если ошибка ожидаема (например, часто отсутствуют ключи в словаре), лучше использовать `dict.get()` или проверку `if key in dict`.
